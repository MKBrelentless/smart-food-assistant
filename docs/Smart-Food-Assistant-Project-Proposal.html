Abstract

The Smart Food Assistant is a full-stack system that helps users determine whether food items are fresh or spoiled using a deep learning model, and provides nutrition-related analyses to guide healthier choices. This proposal outlines the motivation, objectives, architecture (Flask AI service, Node/Express backend, and React frontend), methodology, implementation plan, evaluation approach, and projected resources. The project targets improved food safety, reduced waste, and user-friendly dietary insights.

Introduction

Background & Problem Statement

Food waste and foodborne illnesses have significant health and economic impacts. Consumers often lack quick, reliable tools to assess freshness, leading to premature disposal or consumption of unsafe food. Modern computer vision and web technologies enable a practical solution for image-based freshness assessment and personalized nutrition guidance.

Objectives

  • Provide real-time assessment of food freshness from an image.
  • Offer evidence-based nutrition and diet analysis features.
  • Store user scan history for trend analysis and accountability.
  • Deliver a simple, responsive web interface accessible on desktop and mobile.

Scope & Deliverables

  • Flask AI service exposing prediction and nutrition endpoints.
  • Node/Express backend with authentication, history, and AI integration.
  • React frontend for login/register, scanning, history, and dietary tools.
  • Deployment-ready documentation and API references.

Related Work / Literature Review

Image-based food quality assessment is common in agriculture and retail. Convolutional neural networks (CNNs) excel in classification tasks, including freshness detection. Nutrition guidance often uses food composition databases and heuristic or rule-based models. This project integrates these ideas into a cohesive, user-facing application.

Methodology & System Design

System Architecture

  • AI Service (Flask + TensorFlow, port 5001): Hosts the trained model and nutrition analysis endpoints.
  • Backend API (Node/Express + MongoDB, port 3001): Authentication, authorization (JWT), scan proxy/storage, and history.
  • Frontend (React, port 3000): User interface integrating with both services.

Components

  • AI Service: /predict, /analyze-diet, /analyze-meal, /health-recommendations. Loads model from food_model.keras.
  • Backend API: /auth (login/register), /api/food/scan (forward to AI and store), /api/food/history (retrieve user scans).
  • Frontend: Pages for Login, Register, Dashboard, Scan, History, and Diet Analysis; integrates via src/api.js.

Data Sources & Model

  • Dataset under datasets/mydataset with labeled fresh/spoiled categories.
  • TensorFlow/Keras CNN trained; predictions simplified to Fresh/Spoiled with recommendations.
  • Model artifact: ai-service/food_model.keras; training scripts in ai-service/*.py.

Security, Privacy & Compliance

  • JWT-based auth for protected routes.
  • Validate uploads, enforce size/type limits, define retention policy for images.
  • Move secrets to environment variables; do not commit secrets to source control.

Implementation Plan

Technology Stack

  • Frontend: React, Axios
  • Backend: Node.js, Express, Mongoose, Multer, JWT
  • AI Service: Python, Flask, TensorFlow/Keras, NumPy, Pillow
  • Database: MongoDB

Timeline & Milestones

  1. Week 1–2: Requirements, dataset curation, project scaffolding.
  2. Week 3–4: Model training and evaluation; Flask service with /predict.
  3. Week 5–6: Backend auth and scan/history APIs; AI integration.
  4. Week 7–8: Frontend pages and UX; end-to-end integration.
  5. Week 9: Testing, bug fixes, performance tuning.
  6. Week 10: Documentation and deployment plan.

Risks & Mitigations

  • Model accuracy: dataset augmentation, retraining, threshold tuning.
  • Performance: server-side inference, caching, optimized preprocessing.
  • Network/CORS: proper CORS config, environment-driven base URLs.
  • Security: rate limiting, validation, secret management via .env files.

Evaluation & Testing

  • Model: accuracy, precision/recall, confusion matrix on validation set.
  • API: unit/integration tests for auth, scan, history.
  • Frontend: smoke tests; manual flows for login, upload, history.
  • KPIs: prediction accuracy ≥ target, p95 latency for /predict under target, low error rate on happy-path flows.

Budget & Resources

  • Compute: local dev machine; optional GPU for training.
  • Software: open-source stack (Node, Python, MongoDB, React, TensorFlow).
  • Cloud (optional): hosting for API/AI service, object storage, managed MongoDB.
  • Personnel: 1–2 developers over ~10 weeks (scope-dependent).

Sustainability & Maintenance

  • Versioning, CI for builds/tests.
  • Service logging/monitoring; model drift checks.
  • Docs maintained with code; periodic dependency updates.

Conclusion

The Smart Food Assistant leverages ML and web technologies to address food safety and nutrition needs. The modular design and plan provide a feasible path to a robust product with room for future enhancements (expanded categories, personalization, deployment hardening).

Appendices

Appendix A: API Summary

  • Backend: POST /auth/register, POST /auth/login, POST /api/food/scan, GET /api/food/history
  • AI Service: POST /predict, POST /analyze-diet, POST /analyze-meal, POST /health-recommendations

Appendix B: Environment Configuration

  • Backend .env: MONGO_URI, JWT_SECRET, PORT
  • Frontend .env: REACT_APP_NODE_BASE_URL, REACT_APP_AI_BASE_URL
  • AI Service: optional PORT via environment

Appendix C: Setup & Run Instructions

  • AI Service: cd ai-service; create venv; install deps; ensure food_model.keras; python app.py.
  • Backend: cd backend; npm install; npm run dev.
  • Frontend: cd frontend; npm install; npm start.
